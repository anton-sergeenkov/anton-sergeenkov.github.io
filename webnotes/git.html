Базовые операции


$ git clone                               # создание локальной копии удаленного git-репозитория (выполняется 1 раз)
$ git add .                               # добавляет содержимое рабочей директории в индекс (staging area) для последующего коммита. сборка слепка коммита
$ git push                                # отправка изменений на удаленный git-репозиторий
$ git pull                                # получить изменения из удаленного git-репозитория в локальный репозиторий

Commit


$ git commit                              # сохранение изменений в локальном репозитории. Берёт все данные, добавленные в индекс с помощью git add, и сохраняет их слепок во внутренней базе данных, а затем сдвигает указатель текущей ветки на этот слепок
$ git commit -a                           # добавление всех изменений в индекс без использования git add
$ git commit -m "comment"                 # передача сообщения коммита без запуска полноценного редактора

Branches


$ git checkout -b features                # создание новой ветки "features"
$ git branch                              # текущая ветка
$ git checkout master                     # переключение на ветку "master"
$ git push --set-upstream origin features # 
$ git branch --delete features            # удаление ветки

Diff


$ git diff                                # вычисление разницы между любыми двумя Git деревьями
$ git diff                                # разница между рабочей директорией и индексом
$ git diff --staged                       # разница между индексом и последним коммитом
$ git diff master branchB                 # разница между любыми двумя коммитами
$ git difftool                            # запускает внешнюю утилиту сравнения для показа различий в двух деревьях, на случай если вы хотите использовать что-либо отличное от встроенного просмотрщика git diff


$ git status                              # показывает состояния файлов в рабочей директории и индексе: какие файлы изменены, но не добавлены в индекс; какие ожидают коммита в индексе
$ git init                                # 
$ git reset                               # отмена изменений. Она изменяет указатель HEAD и, опционально, состояние индекса
$ git rm                                  # удаление файлов из индекса и рабочей директории
$ git mv                                  # удобный способ переместить файл, а затем выполнить git add для нового файла и git rm для старого
$ git clean                               # удаление мусора из рабочей директории. Это могут быть результаты сборки проекта или файлы конфликтов слияний

Gitflow

  • Gitflow - модуль управления ветками. Есть две основные ветки: master и develop
  • gitflow-avh – инструмент для работы с Gitflow

master  # содержится тот же код, что и в рабочей (продакт) версии проекта
develop # происходит вся работа
feature # создаются в процессе работы над develop, их может быть неограниченное количество
release # подготовка к новому релизу проекта
hotfix  # срочное исправление багов, найденных, например, на продакте

Рабочий процесс в Gitflow

  1. Создается репозиторий
  2. Репозиторий инициализируется
  3. Начинается работа на ветке develop
  4. Возникает необходимость опробовать новую штуку – создается feature-ветка и делаются коммиты
  5. Закончив работу на feature-ветке, вы сливаете ее с develop
  6. Если вы довольны текущей версией, но хотите продолжить работу, создается ветка release, куда перемещается текущая версия. Правка багов будет происходить на этой же ветке
  7. Когда с веткой release покончено, время слить ее в master и продолжить работу с develop
  8. Кроме того, этот момент можно отметить на master-ветке

Системы управления версиями

  • Централизованная система управления версиями (Version Control System, VCS или Revision Control System) - позволяет хранить несколько версий одного и того же документа, при необходимости возвращаться к более ранним версиям, определять, кто и когда сделал то или иное изменение, и многое другое
  • Такие системы наиболее широко используются при разработке программного обеспечения для хранения исходных кодов разрабатываемой программы
  • Традиционные системы управления версиями используют централизованную модель, когда имеется единое хранилище документов, управляемое специальным сервером, который и выполняет бо́льшую часть функций по управлению версиями
  • Контроль версий означает что вы храните все версии редактируемых документов и можете вернуться к любой сохраненной версии в любой момент времени
  • Примеры: SVN, Team Foundation Server
  • Распределённые системы управления версиями (Distributed Version Control System, DVCS) - используют распределённую модель вместо традиционной клиент-серверной. Они, в общем случае, не нуждаются в централизованном хранилище: вся история изменения документов хранится на каждом компьютере, в локальном хранилище, и при необходимости отдельные фрагменты истории локального хранилища синхронизируются с аналогичным хранилищем на другом компьютере. В некоторых таких системах локальное хранилище располагается непосредственно в каталогах рабочей копии
  • Когда пользователь такой системы выполняет обычные действия, такие как извлечение определённой версии документа, создание новой версии и тому подобное, он работает со своей локальной копией хранилища. По мере внесения изменений, хранилища, принадлежащие разным разработчикам, начинают различаться, и возникает необходимость в их синхронизации. Такая синхронизация может осуществляться с помощью обмена патчами или так называемыми наборами изменений (англ. change sets) между пользователями.
  • С точки зрения пользователя распределённая система отличается необходимостью создавать локальный репозиторий и наличием в командном языке двух дополнительных команд: команды получения репозитория от удалённого компьютера (pull) и передачи своего репозитория на удалённый компьютер (push). Первая команда выполняет слияние изменений удалённого и локального репозиториев с помещением результата в локальный репозиторий; вторая — наоборот, выполняет слияние изменений двух репозиториев с помещением результата в удалённый репозиторий
  • Примеры: Git, Mercurial
  • Git - распределенная система контроля версий (version control system - VCS)
